# 커피 주문 서비스

포인트로 커피 메뉴를 주문하고, 주문 이벤트를 트랜잭션 아웃박스로 안전하게 전송하는 Spring Boot API입니다.

## 주요 기능

- 메뉴 목록과 최근 7일 인기 메뉴 TOP 3 조회
- 사용자 포인트 충전 및 잔액 기반 주문 결제
- 포인트 차감·주문 저장·아웃박스 저장의 단일 트랜잭션 처리
- 커밋 후 주문 이벤트 전송 및 실패 이벤트 재시도
- 예상하지 못한 예외의 상세 정보를 숨기는 공통 500 오류 응답
- 인기 메뉴 집계와 메뉴 정보를 한 번에 조회해 불필요한 N+1 조회 제거
- PostgreSQL/Flyway 기반 공유 저장소와 다중 인스턴스 동시성 고려

- Java 21, Spring Boot 4.1

## 실행 방법

### 1. PostgreSQL 실행

```bash
docker compose up -d
```

기본 연결 정보는 `jdbc:postgresql://localhost:5432/coffee_order_service`이며, 사용자와 비밀번호는 모두 `coffee`입니다.
필요하면 아래 환경 변수로 변경할 수 있습니다. `application.properties`에서 설정할 때는 해당 환경 변수의 점(`.`)·하이픈(`-`) 표기 프로퍼티 이름을 사용합니다.

| 환경 변수 | 기본값 | 설명 |
| --- | --- | --- |
| `DB_URL` | `jdbc:postgresql://localhost:5432/coffee_order_service` | 데이터베이스 JDBC URL |
| `DB_USERNAME` | `coffee` | 데이터베이스 사용자 |
| `DB_PASSWORD` | `coffee` | 데이터베이스 비밀번호 |
| `SCHEDULER_POOL_SIZE` | `2` | 스케줄러 스레드 수 |
| `OUTBOX_RETRY_DELAY_MS` | `1000` | 아웃박스 재시도 간격(ms) |
| `OUTBOX_SCHEDULING_ENABLED` | `true` | 아웃박스 재시도 스케줄러 활성화 여부 |

### 2. 애플리케이션 실행

```bash
./gradlew.bat bootRun
```

### 3. 테스트 실행

```bash
./gradlew.bat test
```

테스트는 PostgreSQL 호환 모드의 H2와 Flyway 마이그레이션을 사용합니다.

## 데이터 일관성 원칙

- 포인트 차감은 조건부 `UPDATE`로 수행해 잔액이 음수가 되지 않습니다.
- 주문 성공 시 포인트 차감, 주문, 아웃박스 이벤트가 함께 커밋됩니다.
- 주문과 같은 트랜잭션에서 저장하는 주문·아웃박스 이벤트에는 하나의 업무 시각을 기록합니다. 같은 주문에서 나온 두 기록의 기준 시간이 달라지지 않게 하기 위함입니다.
- 이벤트 전송은 DB 커밋이 끝난 뒤 시작하며, 실패하면 아웃박스 이벤트를 재시도 상태로 돌립니다.
- 아웃박스 상태 갱신은 이벤트 ID와 클레임 시도 번호를 함께 확인해, 만료된 이전 처리자가 최신 상태를 덮어쓰지 못하게 합니다.
- 아웃박스는 **최소 한 번(at-least-once)** 전달을 전제로 합니다. 실제 외부 클라이언트를 연동할 때는 같은 이벤트가 여러 번 도착해도 한 번만 처리되도록 아웃박스 이벤트 ID를 멱등 키로 함께 전달해야 합니다.

## 용어 한눈에 보기

| 용어 | 쉬운 설명 |
| --- | --- |
| 트랜잭션 | 여러 DB 작업을 모두 성공시키거나 모두 취소하는 하나의 작업 단위 |
| 아웃박스 | 주문이 성공한 사실을 DB에 먼저 저장해 두고, 이후 외부 시스템으로 안전하게 전달하기 위한 기록 |
| 클레임 | 여러 서버 중 한 서버만 이벤트를 처리하도록 처리 권한을 확보하는 과정 |
| N+1 조회 | 목록을 읽은 뒤 항목마다 추가 DB 조회가 반복되는 비효율적인 패턴 |
| 멱등 처리 | 같은 요청이나 이벤트가 여러 번 들어와도 결과가 한 번 처리한 것과 같도록 만드는 방식 |
