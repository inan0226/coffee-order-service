# Spring 과제 통합 리뷰

## 범위 선택

- 모든 리뷰에서 **핵심 과제와 테스트**를 점검한다.
- controller, DTO, 오류 코드 또는 API 명세를 변경했으면 **API 계약**을 추가한다.
- SQL, entity, repository, Flyway를 변경했으면 **데이터베이스**를 추가한다.
- 트랜잭션, 포인트, 아웃박스, 동시성, 외부 전송을 변경했으면 **안전성·관측**을 추가한다.
- 제출, 병합 또는 배포 직전이면 **릴리스**를 추가한다.
- 변경과 무관한 항목은 점검하지 않았다고 명시하고, 추측성 개선을 요구하지 않는다.

## 핵심 과제와 테스트

1. 요구사항
   - 커피 메뉴 목록 조회 API가 있다.
   - 포인트 충전 API가 있다.
   - 커피 주문 및 포인트 결제 API가 있다.
   - 성공한 주문은 Mock 데이터 플랫폼 이벤트를 전송한다.
   - 인기 메뉴 API는 최근 7일 기준 TOP 3를 반환한다.

2. 정확성
   - 포인트는 음수가 될 수 없다.
   - 잘못된 충전 금액은 거절된다.
   - 존재하지 않는 메뉴는 처리된다.
   - 실패한 주문은 주문을 생성하지 않는다.
   - 실패한 주문은 이벤트를 발행하지 않는다.
   - 인기 메뉴는 성공 주문만 집계한다.

3. 설계
   - controller는 얇게 유지된다.
   - service가 비즈니스 규칙을 가진다.
   - request/response DTO를 사용한다.
   - Entity를 controller 응답으로 직접 노출하지 않는다.
   - 시간 범위 로직은 테스트 가능하다.

4. 테스트
   - service 테스트가 핵심 규칙을 검증한다.
   - 필요한 경우 API request/response 형태를 controller 테스트로 검증한다.
   - Mock 이벤트 전송을 검증한다.
   - 최근 7일 경계값을 테스트하고 실제 현재 시간에 의존하지 않는다.

## API 계약

- controller 매핑, request 제약 조건, response 필드, HTTP 상태, `ErrorCode`, `docs/API_SPEC.md`를 비교한다.
- 엔티티나 내부 예외 메시지를 API 응답으로 노출하지 않는다.
- 필드 삭제·이름 변경, 상태 코드 변경, 검증 강화는 호환성 변경으로 보고한다.
- 입력 처리를 변경했다면 성공·실패·빈 검증을 API 회귀 테스트로 확인한다.

## 데이터베이스

- 마이그레이션은 새 순방향 전용 `V<version>__<description>.sql` 파일로 추가한다. 기존 파일을 수정하지 않는다.
- null·기본값·백필·유니크 제약·인덱스의 쓰기 비용을 검토한다. 파괴적 DDL은 승인 없이는 제안만 한다.
- 인덱스는 실제 필터·조인·그룹화·정렬 쿼리와 대조하고, H2 테스트 프로필에서 적용 가능한지 확인한다.
- `user_points` 원자적 갱신과 아웃박스 상태·선점 소유권을 보존한다.

## 안전성·성능·관측

- 포인트 차감, 주문 저장, 아웃박스 저장은 하나의 트랜잭션이어야 하며, 외부 호출 중 DB 잠금을 잡지 않는다.
- 조건부 SQL 갱신을 읽기-수정-쓰기로 바꾸지 않는다. 동시성 변경은 `spring-concurrency-load-test`로 반복 검증한다.
- N+1, 무제한 조회, 비결정적 정렬, 반복 집계만 변경 범위에서 확인한다.
- 로그에는 주문·아웃박스 같은 비밀이 아닌 상관관계 ID와 결과 상태만 남기고, 토큰·인가 헤더·자격 증명·전체 요청 본문·스택 세부 정보는 남기지 않는다.
- 재현 가능한 실패는 `spring-diagnostic-journal`에 비식별화해 기록한다.

## 릴리스

- `git diff --check`와 집중 테스트를 실행하고, 제출 또는 병합 전에는 전체 Gradle 테스트를 실행한다.
- 변경한 마이그레이션, API 명세, 테스트가 함께 갱신됐는지 확인한다.
- 의도된 작업 트리 변경은 경고로만 보고하고, 관련 없는 변경을 되돌리지 않는다.

## 리뷰 출력 형식

아래처럼 간결하게 출력한다.

```text
범위: 핵심 과제, API 계약, 데이터베이스, 안전성·성능·관측, 릴리스 중 점검한 항목

요구사항 및 설계:
- ...

발견 사항:
- [P1|P2|P3] 파일:줄 — 문제, 영향, 권장 변경

테스트:
- ...

미점검 또는 제한사항:
- ...
```

## 제출 게이트

아래 조건을 만족하지 못하면 과제가 완료됐다고 말하지 않는다.

- 네 가지 요구사항 그룹이 모두 구현되어 있다.
- 전체 테스트가 통과한다.
- 의도적으로 남긴 제한사항이 있다면 명확히 보고한다.
