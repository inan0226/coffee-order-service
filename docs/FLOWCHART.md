# 주문 처리 플로우

```mermaid
flowchart TD
    A[POST /api/orders] --> B{요청 검증 통과?}
    B -- 아니오 --> X[400 INVALID_REQUEST]
    B -- 예 --> C{메뉴 존재?}
    C -- 아니오 --> Y[404 MENU_NOT_FOUND]
    C -- 예 --> D[조건부 UPDATE로 포인트 차감]
    D --> E{차감 성공?}
    E -- 아니오, 사용자 없음 --> Z[404 USER_NOT_FOUND]
    E -- 아니오, 잔액 부족 --> AA[400 INSUFFICIENT_POINTS]
    E -- 예 --> T[Clock에서 업무 시각 1회 확보]
    T --> F[트랜잭션 내 주문 저장<br/>ordered_at = 업무 시각]
    F --> G[같은 트랜잭션에 아웃박스 이벤트 저장<br/>created_at = 같은 업무 시각]
    G --> H[커밋]
    H --> I[201 Created 응답]
    H --> J[AFTER_COMMIT 이벤트]
    J --> K[아웃박스 이벤트 클레임]
    K --> L[외부 데이터 플랫폼 전송]
    L --> M{전송 성공?}
    M -- 예 --> N{클레임 시도 번호가 현재 값과 같은가?}
    N -- 예 --> Q[SENT 상태 기록]
    N -- 아니오 --> R[상태를 덮어쓰지 않고 로그 기록]
    M -- 아니오 --> O{클레임 시도 번호가 현재 값과 같은가?}
    O -- 예 --> S[PENDING 상태로 재시도]
    O -- 아니오 --> R
    S --> P[스케줄러가 재클레임]
    P --> L
```

## 포인트 충전 동시성

1. 기존 포인트 행은 단일 `UPDATE`로 증가합니다.
2. 행이 없으면 새 행을 삽입합니다.
3. 첫 충전이 동시에 들어와 중복 키 오류가 발생하면, 실패한 트랜잭션을 종료하고 새 트랜잭션에서 기존 행 증가를 재시도합니다.

## 아웃박스 상태

| 상태 | 의미 |
| --- | --- |
| `PENDING` | 아직 전송하지 않았거나 재시도가 필요한 이벤트 |
| `PROCESSING` | 한 인스턴스가 전송 권한을 클레임한 이벤트 |
| `SENT` | 외부 전송 성공을 기록한 이벤트 |

클레임 시도 번호를 조건부 갱신에 함께 사용해, 만료된 이전 처리자가 현재 처리자의 상태를 덮어쓰지 못하도록 합니다.
